# Качество кода — стандарты senior-ревью

Мера качества: изменение выглядит так, будто его написал самый аккуратный постоянный автор этого проекта. Не «самый умный код», а самый уместный.

## Согласованность с проектом

- Перед правкой прочитать соседний код: имена, стиль, идиомы, как обрабатываются ошибки, как устроены соседние хендлеры/модули. Новый код читается как продолжение существующего.
- Та же плотность комментариев, что вокруг: в проекте без комментариев не разводить эссе, в задокументированном — не бросать голые функции.
- Схема БД, названия таблиц/полей, формат ответов API — сверить с уже существующими, не изобретать параллельную конвенцию.

## Комментарии

Комментарий пишется только про **ограничение, которого не видно из кода**: «Telegram отменяет платёж, если не ответить за 10 секунд», «BigInteger — tg_id не влезает в int32». Запрещены комментарии:

- пересказывающие следующую строку (`# увеличиваем счётчик`);
- объясняющие ревьюеру, почему правка верна (`# исправлено: раньше тут был баг`) — это разговор с проверяющим, а не с будущим читателем;
- отмечающие происхождение (`# добавлено по запросу`, `# новый код`).

## Простота и объём изменения

- Простое практичное решение важнее «правильного по учебнику». Функция лучше класса, если состояния нет; словарь лучше иерархии классов, если вариантов три.
- Абстракции «на вырост» запрещены: не строить плагинную систему для двух случаев, не выносить в конфиг то, что никогда не меняется.
- Менять только то, о чём просили. Попутный рефакторинг, переименования «до кучи», перестановка импортов в незатронутых файлах — запрещены: раздувают диф и прячут суть. Заметил проблему рядом — упомяни в отчёте, не трогай молча.
- Удалять код можно и нужно, когда правка делает его мёртвым — но именно его, а не «заодно и это».

## Полнота

- Целый файл или целая функция/класс — то, что можно вставить и запустить. Внутри изменённого фрагмента никаких `# ... остальное без изменений`, `...`, `pass  # тут логика`.
- Новые зависимости — точной командой установки и с версией, если она важна: `pip install "apscheduler<4"`.
- Импорты в выдаваемом коде полные: читатель копирует файл — файл работает.

## Надёжность

- Каждый внешний вызов (Telegram API, БД, HTTP, файлы) — ответить себе: что случится при ошибке? Пользователь заблокировал бота, сеть моргнула, строки нет в БД. Молчаливое падение хендлера запрещено; молчаливое проглатывание `except Exception: pass` — тоже (минимум лог).
- Идемпотентность там, где возможен повтор: двойное нажатие кнопки, повторная доставка апдейта, повторный `successful_payment`.
- Типы данных: `tg_id` → `BigInteger`; деньги → `int` в минимальных единицах (копейки/звёзды), не `float`; время → UTC-aware `datetime`, конвертация в локальное только при показе.

## Безопасность

- Секреты — только из окружения (env-файл вне git, Script Properties); ни в коде, ни в логах, ни в сообщениях об ошибке.
- SQL — только параметризованный; конкатенация строк в запрос запрещена даже «для внутреннего скрипта».
- Всё, что пришло от пользователя: `.strip()`, проверка на пустоту, ограничение длины; при выводе в HTML — экранирование.
- WebApp initData проверяется подписью на бэкенде; вебхук — с `secret_token`; «проверка на фронте» проверкой не является.

## Анти-паттерны

- Псевдокод и заглушки под видом рабочего кода.
- Выдуманные методы, параметры и версии библиотек. Не уверен в сигнатуре — скажи и предложи свериться с документацией.
- Рефакторинг рабочего кода без конкретной причины.
- «Умный» однострочник вместо трёх читаемых строк.
- Копипаста одной и той же логики в третье место — вместо выноса в функцию (двух мест — можно).

# Git — дисциплина версионирования

Git — не архив «на всякий случай», а рабочий инструмент: точки отката, история изменений и канал деплоя. Правила ниже — для типового цикла «пишу локально → пушу на GitHub → на VPS `git pull`».

## Гигиена до первого коммита

`.gitignore` создаётся **до** первого коммита, а не после утечки токена:

```
.env
venv/
__pycache__/
*.pyc
bot.db
*.log
```

- Секрет попал хотя бы в один коммит — он **скомпрометирован**, даже если следующим коммитом файл удалён: история хранит всё. Единственное правильное действие — сменить токен (BotFather → Revoke) / пароль, потом чистить историю.
- Перед каждым коммитом: `git status` и `git diff --staged` — глазами. `git add .` вслепую запрещён: так в репозиторий въезжают `.env`, дампы БД и временные файлы.

## Коммиты

- **Атомарность**: один коммит — одно логическое изменение («добавил оплату Stars», а не «оплата + починил клавиатуру + переименовал модели»). Смешанный коммит нельзя ни откатить, ни понять.
- **Сообщение** отвечает «что и зачем», а не «где ковырялся»: `Add Stars payment with refund support` — хорошо; `fix`, `изменения`, `wip2` — мусор, который через месяц не расшифрует даже автор.
- **Коммит перед экспериментом** — дешёвая точка отката: собрался переделывать рабочий код — сначала закоммить рабочее состояние.
- Забыл файл в только что сделанном коммите — `git commit --amend` (только пока коммит **не запушен**).

## Ветки

- `main` всегда в запускаемом состоянии — именно её тянет сервер.
- Рискованная переделка (миграция схемы, замена библиотеки, большой рефакторинг) — в ветке: `git switch -c feature-x`; влилось и проверилось — в `main`.
- Мелкие правки соло-разработчику можно прямо в `main` — ветка ради ветки не нужна.

## Откат

- Опубликованное (запушенное) изменение откатывается **только** через `git revert <hash>` — новый коммит-отмена, история цела.
- `git reset --hard` — только для локального, ещё не запушенного, и с пониманием, что несохранённая работа исчезнет безвозвратно.
- `push --force` в ветку, которую тянет кто-то ещё (включая сервер), запрещён: чужие/серверные копии ломаются. Крайний случай — `--force-with-lease` и только в свою личную ветку.

## Деплой через git

- На сервере — только `git pull` и рестарт (см. `vps-deploy.md`). **Править код на сервере запрещено**: следующая выкладка молча затрёт правку, или `pull` встанет с конфликтом.
- Пришлось экстренно поправить на сервере — сразу же перенести правку в репозиторий тем же днём.
- Проверка «что уедет на сервер» перед пушем: `git log origin/main..HEAD --oneline` (какие коммиты) и `git diff origin/main --stat` (какие файлы).

## Типовые ситуации

| Ситуация | Действие |
|---|---|
| Нужно срочно переключиться, работа не доделана | `git stash` → потом `git stash pop` |
| Что сломало бота: какая из последних правок? | `git log --oneline -10`, затем `git diff <hash> -- файл` |
| Локальный файл испорчен, вернуть из последнего коммита | `git restore файл` |
| `pull` ругается на локальные изменения на сервере | Понять, откуда они (кто-то правил на сервере — см. выше), не делать бездумно `checkout .` |

## Анти-паттерны

- `git add .` не глядя; коммит `.env`, `venv/`, дампов БД.
- Сообщения «fix», «wip», «правки» — история, по которой нельзя искать.
- Один гигантский коммит в конце дня со всем подряд.
- Правки кода на сервере мимо репозитория.
- `push --force` в общую ветку; `reset --hard` на запушенном.
- Удалить утёкший токен из файла и считать проблему решённой (токен уже в истории — только revoke).

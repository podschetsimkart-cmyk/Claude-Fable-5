# Стиль взаимодействия с владельцем — из реальной практики

Правила ниже — не предположения: каждое выведено из прямых требований пользователя в рабочих сессиях (проекты Zadachka, SportTeamBot, as-polis). При конфликте с общими правилами скилла побеждает этот файл.

## Краткость — жёсткое требование

Прямая цитата: «я не читаю тонны твоих сообщений, дай конкретику и коротко».

- Первой строкой — действие или ответ. Без предысторий, обзоров вариантов и пересказов сделанного по шагам.
- Детали не вываливать, а предлагать: «скажи, если нужны детали».
- Таблицы и стены текста — только если попросил.

## Вопросы — по одному и опросом

Прямое требование (2026-06-20): уточнения задавать **строго по одному за раз** и **в формате опроса** с 2–4 готовыми вариантами (в Claude Code — AskUserQuestion), а не списком из 3–5 вопросов в чат.

- Варианты должны покрывать вероятные ответы, чтобы редко приходилось печатать своё.
- Нужны несколько уточнений — первый вопрос, дождаться ответа, потом следующий.

## Новая фича — протокол обдумывания до кода

Прямое требование: «сначала обдумываешь, сравниваешь с реальным кодом, говоришь противоречия, обозначаешь лучший вариант, задаёшь вопросы, потом идёшь делать код».

1. Сопоставить идею с фактическим текущим кодом (не с представлением о нём).
2. Назвать противоречия — с существующим поведением или внутри самой идеи.
3. Предложить лучший вариант.
4. Задать улучшающие вопросы (по одному, опросом — см. выше).
5. Только после ответов — писать код.

## «Простыми словами» — без кода

Частая просьба: «расскажи простыми словами», «давай без кодов простыми словами что видят игроки». Это запрос объяснения, а не реализации: отвечать обычным языком глазами пользователя/игрока, **ни одного блока кода**, никаких имён функций и файлов без необходимости.

## Вход может быть шумным

Сообщения приходят с голосовой диктовки: опечатки («перезаупсти», «сейччас»), нет знаков препинания, склейка мыслей. Интерпретировать по смыслу и делать; переспрашивать только если реально возможны два разных действия, а не из-за опечатки.

## Завершение работы — файлы + строка деплоя

Там, где пользователь деплоит сам со своего ПК (SportTeamBot и подобные):

- НЕ предлагать git commit/push, сборку ZIP, scp/ssh/systemctl — всё это он делает руками. Прямые цитаты: «git мне больше не нужен будет… сам сделаю», «архив не нужно теперь делать я сам сделаю с ПК».
- Концовка ответа: изменённые файлы + кратко что менялось, и последней строкой — верхнеуровневые папки для заливки и нужен ли рестарт. Формат: `**Деплой:** app/ + webapp/ + рестарт бота` (или «рестарт не нужен»).

## Один источник правды по проекту

В больших проектах — единственный актуальный справочник (например, `HANDOFF.md`: журнал версий + схема БД + паттерны + роадмап). Правила:

- Нетривиальный вопрос по проекту — сначала читать этот файл, а не заново исследовать код.
- Завершил блок работ — обновил журнал и таблицу версий в нём.
- НЕ плодить параллельные документы (второй «мастер-план» неизбежно отстаёт и врёт — проверено: удалены как устаревшие).

## Закрытая тема — закрыта

Если пользователь явно попросил не поднимать какую-то тему — не поднимать её больше ни в каком виде, даже из лучших побуждений. Вернуться к ней можно только если он сам спросит.

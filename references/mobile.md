# Мобильные приложения — iOS и Android с Windows-машины

Для запросов «сделай приложение», «выложить в App Store / Google Play». Контекст владельца: Windows (Mac нет), готовые FastAPI-бэкенды от Mini Apps, соло-разработка.

## Шаг 0 — а нужно ли приложение

Сначала честно проверить, не хватит ли **Telegram Mini App** (уже умеем, `telegram-miniapp.md`): не нужны магазины, ревью Apple, $99/год и сборка под iOS; обновления мгновенные. Store-приложение оправдано, когда нужны: пуши без Telegram, работа оффлайн, доступ к железу (камера/геолокация в фоне), аудитория вне Telegram, присутствие в магазинах как канал. Если ничего из этого — предложить Mini App и остановиться.

## Рекомендованный стек: Flutter

Один код → iOS + Android (+ web). Почему для владельца: сборка Android полностью на Windows; декларативный UI, который Claude пишет надёжно; горячая перезагрузка.

Альтернатива — Expo/React Native: выбрать, если критичны веб-экосистема JS или OTA-обновления без ревью (EAS Update). Не выбирать нативную пару Swift+Kotlin: двойная работа не для соло.

## Проблема iOS-без-Mac и её решение

Собрать и подписать iOS-приложение на Windows **нельзя** — только macOS-окружение. Рабочие пути:

1. **Облачная сборка** (основной): Codemagic (родной для Flutter, бесплатный тариф есть) или GitHub Actions с macos-раннером. Пуш в репозиторий → облако собирает `.ipa` → выгружает в TestFlight.
2. Разработка и отладка при этом — на Windows: Android-эмулятор + Chrome (Flutter web) покрывают 95% работы; iOS-специфику проверять через TestFlight на реальном iPhone.

## Архитектура: приложение = ещё один клиент вашего API

Бэкенд уже есть — FastAPI от Mini App. Приложение ходит в те же эндпоинты:

- Не дублировать логику в приложении — вся бизнес-логика на сервере, приложение показывает и отправляет.
- Авторизация: логин через Telegram (кнопка «Войти через Telegram» → deep link в бота → бот выдаёт одноразовый код) либо классика (телефон/e-mail + JWT). Для компаньона к боту — телеграмный вход естественнее.
- Пуши: Firebase Cloud Messaging (работает и на iOS через APNs) — токен устройства хранить на бэкенде рядом с tg_id.
- Модели данных в Dart генерировать по образцу ответов API, не выдумывать.

## Публикация — цифры и сроки

| | Google Play | App Store |
|---|---|---|
| Аккаунт | $25 один раз | $99/год |
| Ревью | часы–1 день | 1–3 дня, строже |
| Тест-канал | Internal testing | TestFlight |

Google Play с 2024 требует для личных аккаунтов закрытый тест с 12+ тестерами перед продакшеном — закладывать 2 недели. Приватность: обеим площадкам нужна privacy policy (страница на as-polis.ru подойдёт).

## Минимальный путь до первого приложения

1. `flutter create app_name` → каркас; проверить запуск на Android-эмуляторе.
2. Экран логина + один рабочий экран с данными из вашего API.
3. Android: `flutter build apk` → поставить на свой телефон, погонять неделю.
4. iOS: подключить Codemagic → TestFlight → погонять на iPhone.
5. Только после живого использования — магазины (иконки, скриншоты, описания — по `marketing.md`, интерфейс — по `taste.md`).

## Грабли

- Не начинать с «сразу в оба магазина» — сначала APK себе на телефон; половина идей умирает на этой стадии бесплатно.
- Секреты (ключи API) не зашивать в приложение — apk распаковывается; чувствительное держит бэкенд.
- Версии: `pubspec.yaml` `version: 1.0.0+1` — build number (+1) обязан расти с каждой загрузкой в магазин.
- Отклонение Apple — норма первой попытки: чаще всего за пустые экраны без данных, логин без гостевого режима, отсутствие ссылки на privacy policy. Читать причину и чинить, не спорить.
- На Windows пути в командах Flutter — через `py`-стиль не работает: Flutter свой CLI, просто `flutter <команда>` в PowerShell (после установки Flutter SDK и Android Studio).

# Кибербезопасность — защита ботов, Mini Apps и VPS

Только оборона: харденинг, аудит, реакция на утечки. Дополняет разделы безопасности в `code-quality.md`, `telegram-miniapp.md`, `vps-deploy.md`, `git.md` — здесь сводный чеклист и то, чего там нет.

## Чеклист аудита (прогонять по запросу «проверь безопасность»)

**Секреты**
- Токены/ключи только в env; поиск по коду: `grep -rE "[0-9]{8,10}:[A-Za-z0-9_-]{35}|sk-[a-zA-Z0-9-]{20,}"` — должен быть пуст.
- `.env` в `.gitignore` И не в истории git (`git log --all --full-history -- .env`).
- Секреты не в логах, не в сообщениях об ошибках пользователю.

**Бот**
- Админ-команды закрыты фильтром по `admin_ids` на всём роутере, не в каждом хендлере отдельно.
- Вебхук — с `secret_token`; без него на URL может постить кто угодно.
- Ввод пользователя ограничен по длине до записи в БД; SQL только параметризованный.
- Callback-данные не доверяются: право на действие проверяется по БД, а не по тому, что кнопка «была показана только админу».

**Mini App**
- `user_id` берётся ТОЛЬКО из проверенного initData (подпись + срок давности), никогда из тела запроса.
- Все мутирующие эндпоинты — за `Depends(auth_user)`; проверить, что ни один не забыт.

**VPS**
- SSH: вход только по ключу (`PasswordAuthentication no` в sshd_config), `PermitRootLogin no`.
- `ufw`: открыты только OpenSSH и Nginx; порт приложения (8080) наружу закрыт — только через nginx.
- `fail2ban` на SSH: `apt install fail2ban` — работает из коробки.
- Обновления: `apt install unattended-upgrades` — патчи безопасности сами.
- Бот работает под своим пользователем без sudo (`User=botuser`, `NoNewPrivileges=true`).

**Зависимости**
- `pip list --outdated` периодически; известные дыры: `pip install pip-audit && pip-audit`.

## Реакция на утечку секрета

Порядок жёсткий, без пропусков:

1. **Отозвать немедленно**: токен бота — BotFather → `/revoke` (бот получает новый токен, старый умирает); API-ключи — в консоли сервиса. Удаление файла/коммита БЕЗ отзыва ничего не решает — история и чужие копии живут вечно.
2. Заменить секрет в `.env` на сервере, рестарт сервиса.
3. Проверить, не успели ли воспользоваться: логи бота на незнакомые команды, `getWebhookInfo` (не увели ли вебхук на чужой URL), история Stars-транзакций.
4. Понять канал утечки и закрыть его (коммит в git, пересланный файл, скриншот, лог).

## Принцип минимальных прав

Каждому компоненту — только то, что нужно: deploy-пользователь умеет рестартить один сервис (`sudoers.d`), бот в БД — не суперпользователь, API-ключ — с минимальными скоупами, если сервис позволяет. Тогда одна утечка не отдаёт всё.

## Чего не делать

- Не писать собственную криптографию и собственную проверку подписи там, где есть готовая проверенная (initData — по официальной схеме).
- Не «прятать» секрет в коде через base64/склейку строк — это не защита.
- Не отключать проверку сертификатов (`verify=False`) «чтобы заработало».
- Не давать боту прав админа канала/группы больше необходимых.

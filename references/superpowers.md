# Superpowers — процессные практики для крупных задач

Дистилляция ключевых практик плагина superpowers, согласованная с остальным скиллом. Применять на задачах **крупнее пары функций** — новая фича, модуль, переделка. На мелких правках эти церемонии не нужны. Пересечения с уже существующими правилами решены так: верификация — `verification.md`, отладка — `debugging.md`, вопросы владельцу — `interaction.md` (по одному, опросом).

## Брейншторм до кода (крупная фича)

Прежде чем проектировать — короткий исследовательский цикл:

1. Прочитать существующий код вокруг будущей фичи (не по памяти).
2. Сформулировать 2–3 варианта реализации с ценой каждого (сложность, риск, что ломается).
3. Применить YAGNI-фильтр: вычеркнуть из вариантов всё «на будущее». Лучший вариант — обычно самый скучный.
4. Показать владельцу рекомендацию + альтернативу одной строкой; спорные развилки — вопросом с вариантами (по одному).

## План с фазами и чекпоинтами

Для задач на несколько сессий/этапов план пишется в файл (например, `PLAN.md` или блок в HANDOFF.md), а не держится в голове. Формат каждой фазы:

```
### Фаза N: <название>
Цель: <одно предложение — что станет возможным>
Шаги: <2-5 конкретных шагов с именами файлов>
Проверка: <команда или действие с ожидаемым результатом>
Готово когда: <бинарный критерий, без «в основном работает»>
```

Правила:
- Фаза заканчивается **работающим состоянием** — бот запускается, тесты зелёные. Никогда не оставлять проект сломанным между фазами.
- После фазы — чекпоинт: отметить в плане сделанное, зафиксировать отклонения от плана (они — норма, молчание о них — нет).
- Обнаружил, что план устарел, — обнови план, а не продирайся через неактуальный.

## TDD-цикл для бизнес-логики

Для логики в `services/` (деньги, лимиты, права, подсчёты) — строгий цикл:

1. **Красный**: написать тест на новое поведение, запустить, **увидеть падение**. Тест, который не падал, ничего не доказывает.
2. **Зелёный**: минимальный код, чтобы тест прошёл. Не больше.
3. **Рефакторинг**: причесать при зелёных тестах.

Когда TDD не нужен: разовые скрипты, правки текстов, хендлеры-однострочники, эксперименты. Не превращать дисциплину в карго-культ.

## Дробление и параллельность

Большую задачу резать на независимые куски и делать их по очереди с проверкой после каждого — а не одним гигантским изменением «всё сразу». В Claude Code независимые исследования (прочитать три части кодовой базы, проверить две гипотезы) можно поручать субагентам параллельно; правки в одни и те же файлы — только последовательно.

## Самоконтроль процесса

Признаки, что процесс сломался и надо остановиться:

- третий раз подряд «попробуй теперь» без новой гипотезы → вернуться к протоколу отладки;
- диф разросся далеко за рамки задачи → откатиться к последнему чекпоинту и продолжить меньшими шагами;
- план и код разошлись → сначала синхронизировать план;
- хочется закомментировать упавший тест «чтобы не мешал» → это сигнал о реальном баге, а не о лишнем тесте.

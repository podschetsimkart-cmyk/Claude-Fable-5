# Fable 5 Style — рабочий цикл и стандарты

Цель: код уровня senior-ревью с первого ответа — запускается как есть, переживает край-кейсы, ничего не выдумано. Скилл не «стиль оформления», а дисциплина, которая это обеспечивает.

## Рабочий цикл (обязательный порядок)

1. **Понять задачу.** Есть неочевидное допущение — прими разумное и укажи его одной строкой вместо лишнего вопроса. Вопрос задавай только если от ответа зависит архитектура.
2. **Открыть справочник** из таблицы ниже — там проверенные паттерны и типовые ошибки. Не пиши по памяти то, что есть в справочнике. Задача на стыке (бот + платежи) — читай оба файла.
3. **План** (для задач больше одной функции): 3–6 строк — какие файлы, что меняется, какие риски. Показать план, потом код.
4. **Код** — целые файлы или целые функции/классы. Никаких заглушек `pass  # тут логика`, `...` и псевдокода под видом кода.
5. **Самопроверка ДО отправки** — отдельный обязательный проход по чеклисту ниже, не формальность.
6. **Выдача этапами**: этап → «как проверить» → что будет дальше. Не вываливать всё сразу.

## Стандарты кода (без исключений)

- Полный файл или полная функция/класс. Внутри изменённого фрагмента никаких `# ... остальное без изменений`.
- Новые зависимости названы точной командой: `pip install ...`.
- Секреты только из окружения (env / Script Properties), никогда в коде и в git.
- `tg_id` → `BigInteger`; деньги → `int` (копейки / звёзды), не `float`; время → UTC-aware `datetime`.
- Пользовательский ввод: `.strip()`, проверка на пустоту, ограничение длины; в SQL — только параметры, в HTML — экранирование.
- Каждый внешний вызов (Telegram API, БД, HTTP) — ответь себе: что случится при ошибке? Молчаливое падение хендлера запрещено.

## Чеклист самопроверки (прогнать честно, по пунктам)

1. **Мысленный запуск**: пройди код от импортов до результата на конкретном входе (не «в целом ок», а конкретное сообщение / строка таблицы / запрос).
2. **Ничего не выдумано**: каждый вызов внешней библиотеки — метод и параметры точно существуют? Не уверен — скажи прямо и предложи свериться с документацией, а не сочиняй сигнатуру.
3. **Край-кейсы**: пустой список, `None`, двойное нажатие кнопки, параллельный апдейт, текст длиннее 4096, пользователь заблокировал бота.
4. **Безопасность**: токены не в коде; initData проверяется на бэкенде; у вебхука `secret_token`; SQL параметризован.
5. **Согласованность**: имена, стиль и схема БД совпадают с уже существующим кодом из диалога.
6. **Границы сделанного**: чего в ответе НЕТ (миграции, деплой, тесты) — сказать явно, а не молчать.

## Протокол отладки

Когда прислали ошибку:

1. Traceback читай снизу вверх: последняя строка — что случилось, первая строка из кода проекта — где.
2. Сформулируй 1–2 гипотезы, проверь самую вероятную. Переписывание всего подряд («shotgun debugging») запрещено.
3. Мало данных — попроси конкретику: полный traceback, версию библиотеки, `print(...)` конкретного значения. Не «пришли весь код».
4. Исправление давай точечно: имя файла + функция целиком.
5. После исправления — одной строкой: почему это была причина, а не симптом.

## Формат ответов

- Русский язык; код, термины и названия — на английском.
- 2–4 предложения пояснений вокруг кода. Без лекций, лишних списков и жирного текста.
- К каждому этапу — «как проверить»: конкретная команда или действие, а не «должно работать».

## Анти-паттерны (запрещено)

- Заглушки и псевдокод под видом рабочего кода.
- Выдуманные методы, параметры, версии.
- «Должно работать» без способа проверить.
- Рефакторинг рабочего кода без конкретной причины; абстракции «на вырост».
- Переусложнение: простое практичное решение важнее «правильного по учебнику».
- Больше одного вопроса за раз.
